# Running Your Own Registry

For enterprises that need to host attestation data for internal packages or mirror the public registry.

## Architecture

The registry is a Next.js application with pluggable storage backends (Cloud Storage, S3, Azure Blob, local filesystem).

## Deployment Options

### Docker

\`\`\`bash
docker pull korext/oss-registry:1.0
docker run -p 3000:3000 \\
  -e STORAGE_BACKEND=s3 \\
  -e STORAGE_BUCKET=acme-korext \\
  -e AWS_ACCESS_KEY_ID=... \\
  -e AWS_SECRET_ACCESS_KEY=... \\
  -e AUTH_PROVIDER=oauth \\
  korext/oss-registry:1.0
\`\`\`

## Configuration

Environment variables:

  STORAGE_BACKEND: cloud-storage | s3 | azure | local
  STORAGE_BUCKET: bucket or path
  STORAGE_CREDENTIALS: as required per backend
  PUBLIC_REGISTRY_URL: for federation (proxies unknown packages to public)
  AUTH_PROVIDER: oauth | saml | token | none
  AUTH_ISSUER_URL: for OAuth/SAML
  AUTH_CLIENT_ID: auth client id
  AUTH_CLIENT_SECRET: auth client secret
  ADMIN_TOKEN: for admin operations

## Client Configuration

Create .supply-check.yaml in your repo:

\`\`\`yaml
registries:
  - name: public
    url: https://oss.korext.com/registry
    priority: 1

  - name: internal
    url: https://korext.acme.com/registry
    priority: 2
    auth:
      type: token
      env: ACME_KOREXT_TOKEN
\`\`\`

The CLI queries in priority order. Internal data overrides public when both exist.

## Federation

Set PUBLIC_REGISTRY_URL on the private registry. Unknown package queries proxy to the public registry. Full coverage of public packages plus internal packages through a single endpoint.

## Authentication

### Token

Simplest option. Tokens are random strings generated by an admin. Stored in environment variables on the client.

### OAuth

Integrates with Okta, Auth0, Azure AD, Google Workspace. Users authenticate via their corporate SSO.

### SAML

Enterprise SSO for on-premises setups.

## Storage Backends

All four backends are tested and production ready in v1.0:

  Cloud Storage: for deployments on Google Cloud
  S3: for AWS deployments (also works with MinIO, Ceph, and other S3-compatible stores)
  Azure Blob: for Azure deployments
  Local filesystem: for air-gapped or small deployments

## Scaling

Horizontal scaling: the registry is stateless (state lives in storage backend). Run multiple replicas behind a load balancer.

Caching: add a CDN in front for read-heavy workloads.
